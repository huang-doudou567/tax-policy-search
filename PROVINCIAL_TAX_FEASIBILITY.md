# 接入全国各省市税务局官网 — 操作难度评估

## 实测结论：不适合直接批量接入，推荐走 WebSearch 间接路由

---

## 实测数据

对 8 个省市税局网站做了三步测试：

### 1. 直连访问

| 省市 | 首页 | WAS5 搜索 |
|------|------|----------|
| 上海 | ✅ HTTP 200 | ✅ 200 (但返回空) |
| 广东 | ✅ HTTP 200 | ❌ 404 |
| 深圳 | ✅ HTTP 200 | ❌ 404 |
| 浙江 | ✅ HTTP 200 | ⚠️ 200 但返回首页 |
| 江苏 | ✅ HTTP 200 | ❌ 404 |
| 北京 | ❌ 连接超时 | ❌ 连接超时 |
| 四川 | ❌ 连接超时 | ❌ 连接超时 |
| 山东 | ❌ HTTP 412 | ❌ HTTP 412 |

```
结论：WAS5 搜索引擎未在各省统一部署。
      每个省的搜索接口、参数、返回格式各不相同。
```

### 2. Bing site: 搜索各省站点

| 省市 | Bing mentions | 可解析链接 |
|------|-------------|-----------|
| 上海 | 26 | 0 |
| 广东 | 26 | 0 |
| 浙江 | 29 | 2 (非政策页) |
| 其余 5 省 | 26 | 0 |

```
结论：Bing 对 .gov.cn 站点的搜索结果链接做了 JS 渲染，
      Python 正则无法直接提取 href。
      但 WebSearch 工具（有浏览器渲染能力）可以正常获取。
```

### 3. 实际内容覆盖

搜索"增值税 小微企业 优惠"：
- 税务总局全国站：15624 条全站搜索命中
- 地方站唯一独有内容：非常少量的本省市具体执行口径/操作指南

## 根本问题

```
┌─────────────────────────────────────────────────────┐
│     省税局 A       省税局 B       省税局 C           │
│   ┌─────────┐  ┌──────────┐  ┌──────────┐          │
│   │自定义搜索│  │WAS5搜索  │  │TRS搜索   │          │
│   │(Java)   │  │(部分可用)│  │(内网)    │          │
│   └─────────┘  └──────────┘  └──────────┘          │
│                                                      │
│  31 个省 + 5 个计划单列市 = 36+ 套不同搜索系统       │
│  无统一 API · 无统一搜索参数 · 无统一返回格式         │
└─────────────────────────────────────────────────────┘
```

每一个省都是一套独立的 CMS 和搜索引擎。要直接对接，需要：
- 每个省写一套 HTML 解析规则
- 每个省维护反爬适配
- 每个省处理登录验证/反爬墙

这是 BAT 量级的工程投入，不适合单个 Skill 做。

---

## 可行方案

### 方案：WebSearch 间接路由（轻量，零新代码）

不是去爬各省网站，而是**让用户指定省份，然后用 WebSearch 工具去搜**：

```python
# 后端新增约 20 行
PROVINCE_SITES = {
    "北京": "site:beijing.chinatax.gov.cn",
    "上海": "site:shanghai.chinatax.gov.cn",
    "广东": "site:guangdong.chinatax.gov.cn",
    "深圳": "site:shenzhen.chinatax.gov.cn",
    ...  # 36 个省市
}

@app.route("/api/local-search", methods=["POST"])
def api_local_search():
    keyword = data["keyword"]
    province = data["province"]
    site_filter = PROVINCE_SITES.get(province, "")
    # 委托给 WebSearch 工具（Claude 内置能力）
    # 返回结构化结果
```

前端改动约 30 行：在筛选区加一个省份下拉框。

### 为什么这个方案可行

| 维度 | 直接爬各省网站 | WebSearch 间接路由 |
|------|-------------|-------------------|
| **开发成本** | 每个省 ~200 行 × 36 省 = ~7,200 行 | **~50 行** |
| **维护成本** | 每个省改版都要修解析 | **零（WebSearch 自动适配）** |
| **实时性** | 依赖爬虫频率 | **实时** |
| **覆盖** | 只有能爬到的 | **所有被搜索引擎索引的页面** |
| **可靠性** | 低（反爬/超时/格式变化） | **高（搜索引擎层兜底）** |

### 本质

```
旧思路：我们去爬每一个省税局网站
   ↓
新思路：让搜索引擎去爬，我们只负责构建精确的搜索 query
```

这和已有的 `site:chinatax.gov.cn` 搜索逻辑完全一致——只是把 `site:` 过滤条件从全国站改成省市站。

## 实施建议

如果要做，分两档：

### 轻量版（1h，~50 行）

```
前端：筛选区加"省份"下拉框（默认"全国"）
  → 选择"上海"后，搜索自动附加 site:shanghai.chinatax.gov.cn
  → 复用已有的 /api/interpretations 端点的 site: 搜索逻辑
```

### 标准版（3h，~100 行）

```
后端：新增 /api/local-search 端点
  → 接受 province + keyword
  → 并行搜索 national + provincial site
  → 返回对比结果（全国政策 vs 地方口径）
前端：搜索结果分两栏（全国法规 | 本地口径）
```

---

## 关键风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 某些省税局网站不被搜索引擎索引 | 搜不到结果 | 标注"该省市暂未索引" |
| 地方性政策也在全国站有副本 | 搜到重复内容 | 按来源 URL 去重（已有 Jaccard） |
| 部分省市内容可能需要内网访问 | 完全搜不到 | 明确告知用户直接访问该省税局官网 |

---

## 一句话总结

**各省市税局没有统一 API，直接爬取不可行。但走 WebSearch 间接路由只需 ~50 行代码，1 小时搞定，零维护成本。** 本质就是把已有的 `site:chinatax.gov.cn` 搜索机制扩展一个省份参数。
