# tax-policy-search 改善审查报告

> 以 vercel-react-native-skills 作为"最佳实践刻度尺"，逐项度量 tax-policy-search 的差距

---

## 度量方法

```
vercel-react-native-skills 的设计特征  →  逐项检查 tax-policy-search 有没有
                                        ↓
                                  有: ✅ (评估质量)
                                  部分有: ⚠️ (评估差距)
                                  没有: ❌ (评估影响 + 建议)
```

---

## 一、结构维度

### 1. 内容组织

| 检查项 | vercel-react-native-skills | tax-policy-search (现状) | 差距 |
|--------|--------------------------|-------------------------|------|
| **入口文件大小** | SKILL.md 122 行（仅索引+速查表） | SKILL.md 292 行（全部内容） | ❌ 太长，Agent 可能跳过关键段落 |
| **详细规则独立文件** | `rules/` 目录，25+ 独立 .md | 无，全部堆在 SKILL.md | ❌ 无法按需加载，也无法单独更新 |
| **编译版文档** | `AGENTS.md` 73,771 行完整规则库 | 无 | ❌ 没有完整参考版 |
| **命名规范** | `category-rule-name.md`（如 `list-performance-virtualize`） | 无规范 | ❌ 规则没有标识符 |

**影响**: SKILL.md 太长 → Agent 大型上下文窗口虽然能容下，但关键规则（如禁止行为）被淹没。Agent 更可能在前 100 行认真遵守，后 200 行逐行衰减。

**建议**: 将第④-⑧节抽取为独立的 `rules/` 目录文件，SKILL.md 只保留入口 + 意图 + 命令速查。

---

### 2. 优先级体系

| 检查项 | vercel-react-native-skills | tax-policy-search | 差距 |
|--------|--------------------------|-------------------|------|
| **规则分级** | CRITICAL / HIGH / MEDIUM / LOW 四级 | 无分级，8 条禁止行为平铺 | ❌ |
| **每个规则标注影响** | "CRITICAL: wrong list performance kills UX" | 无影响说明 | ❌ |
| **违反后果** | 每条规则隐含或明示违反后果 | 无 | ❌ |

**建议**: 8 条禁止行为分级：
```
CRITICAL (生命线): 禁止不搜索就回答、禁止使用训练数据
HIGH (质量线): 禁止省略免责声明、禁止省略时间戳、禁止引用已废止法规
MEDIUM (信任线): 禁止预测稽查结果、禁止给出具体金额建议
LOW (规范线): 禁止代替用户做申报操作
```

### 3. 触发机制

| 检查项 | vercel-react-native-skills | 现状 | 差距 |
|--------|--------------------------|------|------|
| **正向触发** | "When to Apply: building RN components, optimizing lists..." | 有触发关键词列表 ✅ | — |
| **反向触发** | 无（隐含：不做 Web 开发就不触发） | 无 | ❌ |
| **触发信号强弱** | 弱信号也有对应规则（MEDIUM 级别） | 只处理了强信号 | ⚠️ |

**SKILL.md 第 13 行的触发词列表很全，但缺"不触发"场景**：
```
✅ 已有: 增值税、所得税、税务...
❌ 缺失: "我不是问中国税法" → 不激活此 Skill
❌ 缺失: "股权架构"不一定触发税收技能 → 需判是否含税关键词
```

**建议**: 加一个 `## 不触发条件` 段：
```
以下场景不触发本 Skill:
- 纯公司法问题（如股权结构设计）且不含税收关键词
- 境外税法（非中国税法）
- 纯会计做账问题（非税务处理）
```

## 二、内容维度

### 4. 规则格式

vercel-react-native-skills 每条规则包含四段式：

```
规则名称 (例: list-performance-virtualize)
├── 为什么重要 (Why)
├── 错误代码 (Wrong) + 解释
├── 正确代码 (Right) + 解释
└── 补充上下文 (Context)
```

tax-policy-search 的 8 条禁止行为缺少对应的"正确做法示例"：

```
现状:                          vercel 模式建议:
─────────────────────          ──────────────────────
#4 禁止预测稽查结果              #4 禁止预测稽查结果
  (无解释)                       **为什么**: 稽查决定权在税务机关，AI 预测可能误导用户
                                 **错误做法**: ❌ "你这种情况一定会被查"
                                 **正确做法**: ✅ "大额公转私可能触发资金流监控指标，建议确保四流合一"
                                 **上下文**: 金税四期 200+ 指标，该建议基于 XX 法规第 X 条
```

**建议**: 每条禁止行为补上 Why + Wrong 示例 + Right 示例。

---

### 5. 错误示例 vs 正确示例

| 检查项 | vercel | 现状 | 差距 |
|--------|--------|------|------|
| 意图分类有反例吗 | N/A | ❌ 无 | 容易误判 |
| 搜索策略有错误演示吗 | N/A | ❌ 无 | Agent 可能选错参数 |
| 输出格式有错误示例吗 | N/A | ❌ 无 | 回答可能不标准 |

**实际风险场景**：

```
用户问: "我的公司去年亏损，今年盈利，能享受小微企业优惠吗？"

当前意图识别: policy_lookup ❌ 错了
正确意图: eligibility（资格判定）

为什么 misclassify: 因为"小微企业优惠"匹配了 policy_lookup 的"优惠"关键词
但用户实际是在问"我是否符合条件"，不是问"优惠政策是什么"

缺失: 意图分类表里没有反例，Agent 不知道"优惠"在"能否享受"上下文中应该是 eligibility
```

**建议**: 意图表中加一列"易混淆案例"。

---

### 6. 覆盖完整性

| 检查项 | vercel 方式 | 现状 | 差距 |
|--------|-----------|------|------|
| 搜索策略覆盖 | 8 个明确策略 | 8 种常见场景 | ⚠️ 缺边缘场景 |
| 错误处理覆盖 | 每个规则独立 | 1 个集中段（⑧无结果时） | ⚠️ |
| 多数据源降级 | 每条规则有回退 | 无 | ❌ |

**缺失的搜索策略场景**：
```
❌ "我不知道法规叫什么，但知道大概内容" → 无专门策略
❌ "我想对比新旧版本的变化" → 无专门策略
❌ "某个法规的配套实施条例/细则" → 需要 Agent 自己推理
```

**建议**: 搜索策略速查表补 3-4 条高频边缘场景。

## 三、代码维度

### 7. 正确做法示例格式

vercel-react-native-skills 对于代码示例做到了：
- 明确标注 WRONG / RIGHT
- 每个示例有解释
- 代码行内注释标注问题点

tax-policy-search 的命令速查全是正确示例，缺 **"这样做是错的"**：

```bash
# ❌ 当前缺失: 错误示例
python scripts/tax_search.py "增值税" --status 1   # ← 搜已废止，无意义(除非用户明确需要)
python scripts/tax_search.py "" --status 3          # ← 空关键词，必定无结果

# ✅ 当前已有: 正确示例
python scripts/tax_search.py "增值税" --status 3 --size 20
```

**影响**: Agent 凭"直觉"随意组合参数，可能搜出已废止法规。

**建议**: 命令速查表补一个 `## ❌ 常见错误用法` 段，列出 3-5 个典型错误。

---

### 8. 参数选择指导

| 检查项 | vercel | 现状 | 差距 |
|--------|--------|------|------|
| 参数优先级 | CRITICAL 级规则禁止某些写法 | 无 | ❌ |
| 参数默认值原因 | 每条规则解释为什么用这个默认值 | 无 | ⚠️ |
| 参数组合禁忌 | 某些参数不能同时用 | 无 | ❌ |

**实际风险**：
```
Agent 可能会写:
  --search "小微企业" --exact --scope title
  → 过于狭窄，可能搜不到（不是所有法规标题里都有"小微"二字）

应该:
  --search "小微企业" --scope fulltext
  → 正文中也会出现
```

**建议**: 加一个参数组合速查表，标注每个参数的"适用场景"和"不要用的场景"。

## 四、可靠性维度

### 9. 输出格式约束

vercel 的每条规则可独立验证。tax-policy-search 的约束很难验证：

| 约束 | 可自动化验证？ | 差距 |
|------|-------------|------|
| "先搜索后回答" | ⚠️ 无法自动验证 | Agent 可能在回答中声称搜了但没搜 |
| "标注时间戳" | ⚠️ 可验证格式，无法验证真实度 | 可能写假时间戳 |
| "标注法规 ID" | ✅ 可验证是否有 ID | — |
| "不预测稽查" | ❌ 无法自动验证 | 只能靠 Prompt 约束 |
| "不省略免责" | ⚠️ 可验证是否有免责文本 | — |

**建议**: 在 SKILL.md 末尾增加 `## 输出自检清单`，Agent 在回答前逐项自检。

---

### 10. 降级与回退

vercel 的每条规则是独立的——一条失效不影响其他。tax-policy-search 降级策略太粗：

| 降级场景 | 当前做法 | vercel 模式建议 |
|---------|---------|---------------|
| NPC API 超时 | 直接报错 | CRITICAL: 提示用户 + 自动回退 chinatax 搜索 |
| chinatax 搜不到 | 静默跳过 | HIGH: 告知用户该数据源当前不可用 |
| AnySearch 不可用 | 静默跳过 | MEDIUM: 记录但不告知（非关键源） |
| DOCX 解析失败 | 显示"未能获取全文" | MEDIUM: 告知 + 提供 NPC 原页面链接替代 |

**建议**: 每个数据源标注可靠度等级，Agent 在失败时按级别处理（告知 vs 静默降级）。

## 七、一句话总结

> tax-policy-search 在**功能完整性**上已达标，但在 **Prompt Engineering 工程化**上——即让 Agent 不犯错地执行意图——与 vercel-react-native-skills 代表的工业级 Skill 标准差距明显。核心差距不是"功能不够"而是"规则不够精确、例子不够具体、错误路径没有预封堵"。

vercel 的 122 行索引 + 25 条独立规则的模块化模式，比 tax-policy-search 的 292 行单体 SKILL.md 更抗 Agent 误读、更易增量改进。
